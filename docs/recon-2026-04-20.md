# Full-app reconnaissance — 2026-04-20

Written after reading [index.html](../index.html) end-to-end (5,685 lines), [api/digest.js](../api/digest.js), [api/process.js](../api/process.js), and the oauth + task-view docs. Task-display items already covered by [docs/task-view-improvements.md](task-view-improvements.md) are out of scope here.

Today shipped 7 task-view loops (overdue treatment, priority sort, humanized dates, date-window dividers, tint, float). Tasks view is now cohesive. The rest of the app is not.

## Three things to ship NEXT

### 1. Make the digest Mia-safe

[api/digest.js:146](../api/digest.js:146) opens the AI prompt with `"You are STWRD, Vijay Ramdon's personal AI life manager."` and [api/digest.js:101-109](../api/digest.js:101) hardcodes a life-category keyword set stuffed with Vijay-specific tokens (`caliber`, `gne`, `sox`, `serveants`, `honeybook`, `vibo`, `crue`, `jada`). The life-score math, the keyword matching, and the AI's self-identity are all frozen to one user. When Mia gets her 7am digest it will literally greet her as Vijay and score her week against Vijay's job keywords. This is the single biggest blocker to shipping STWRD to anyone else, and it's ~40 lines of work: read `profile.full_name` for the salutation and `life_categories` rows for the keyword set instead of the hardcoded array. Beta-blocker. Do first.

### 2. Rewrite the Today view as "what to do today"

[renderToday at index.html:3562](../index.html:3562) filters `tasks.filter(t => t.due_date === today)` — which on any given day catches maybe 1–3 items, because the AI classifier only extracts `due_date` when a date is literally spoken. Most tasks arrive with `due_date = null`, so Today renders "Nothing due today!" even when the user has 40 active tasks. That makes the second-most-prominent tab a dead end. The fix is to redefine Today as the intersection of *overdue + P1 + explicitly-due-today + AI-picked focus task*, pinned above the rest of the day's likely work — the same signals the morning digest already has. This is the daily-use surface for a non-technical couple; right now it doesn't earn its place in the nav.

### 3. Un-hardcode project colors across drilldowns

Three separate render paths — [showLifeDrilldown:3665](../index.html:3665), [showDrilldown:3715](../index.html:3715), [showCompletedDrilldown:3767](../index.html:3767) — and [api/digest.js:211](../api/digest.js:211) all carry a hardcoded `{ GNE, Caliber, Personal, ServeAnts, Family }` color map. Any custom project Mia creates during onboarding renders in the fallback accent-purple everywhere except the hat-bucket page. The project bucket she sees on the main screen won't visually match the tags she sees in the dashboard drilldown or the digest email — same data, different color, looks like a bug. Replace with one helper that reads `userHats[].color` (already stored in the DB from onboarding's `assignHatPersonality`). Small, unlocks the whole non-Vijay visual experience.

## Ranked observations

### 1. Digest greets every user as Vijay

- **What:** [api/digest.js:146](../api/digest.js:146) hardcodes `"You are STWRD, Vijay Ramdon's personal AI life manager."` in the system prompt; [api/digest.js:101-109](../api/digest.js:101) hardcodes the life-category keyword list.
- **Problem:** First non-Vijay recipient (Mia) gets addressed as Vijay and scored against Vijay's work keywords. Ships unusable for every beta couple.
- **Size:** small
- **Beta impact:** high — this is the entire onramp for a new user
- **Dependencies:** covered by top-three #1

### 2. Today view is mostly empty by design

- **What:** [renderToday:3562](../index.html:3562) filters strictly by `due_date === today`. The AI classifier only extracts a `due_date` when the user explicitly spoke one (see prompt at [index.html:2138-2143](../index.html:2138)), which most tasks don't.
- **Problem:** A user with 30+ active tasks sees "Nothing due today! 🎉" on the tab that's supposed to be their morning anchor. The nav-bar order implies Today is the daily surface; the implementation doesn't deliver that promise.
- **Size:** medium
- **Beta impact:** high — daily-touched surface, determines habit formation
- **Dependencies:** covered by top-three #2

### 3. Project color map forked 4 ways

- **What:** Five places hardcode a GNE/Caliber/Personal/ServeAnts/Family → CSS-var map. [showLifeDrilldown:3665](../index.html:3665), [showDrilldown:3715](../index.html:3715), [showCompletedDrilldown:3767](../index.html:3767), [hatColor:2551](../index.html:2551), and [api/digest.js:211](../api/digest.js:211).
- **Problem:** Custom projects get default accent color everywhere except main bucket page. Onboarding's `assignHatPersonality` already picks a color per project and stores it; nothing reads it back.
- **Size:** small
- **Beta impact:** high — breaks visual consistency for every non-Vijay user
- **Dependencies:** covered by top-three #3

### 4. Snapshot columns hardcoded to seven life areas

- **What:** [renderProgress:4102-4109](../index.html:4102) falls back to seven specific `life_work`, `life_family`, `life_health`, etc. column names. The create-table instructions shown to users at [index.html:4038-4054](../index.html:4038) only include those seven. [saveWeeklySnapshot:3993](../index.html:3993) dynamically builds column names as `life_<name.split(' ')[0]>`, so Mia with a "Marriage"/"Parenting" category writes to `life_marriage` / `life_parenting` — which don't exist in the provisioned schema.
- **Problem:** First weekly snapshot write will silently fail for any user whose categories aren't the exact seven-area default. The Journey tab will show "Day 1 starts now" forever.
- **Size:** medium — requires migration to a key/value `life_scores jsonb` column and renderProgress rewrite
- **Beta impact:** high — Journey is one of five nav tabs and it breaks for anyone non-default
- **Dependencies:** needs a schema migration discussion before coding

### 5. Debug `showToast` calls in production loadPartner

- **What:** [loadPartner:5434-5456](../index.html:5434) emits four user-facing toasts including `"🔍 loadPartner: no currentUserId yet"`, `"🔍 loadPartner found N rows"`, `"✅ Partner loaded: Name"`, `"❌ loadPartner error: ..."`.
- **Problem:** Every sign-in triggers a debug toast barrage. Unprofessional, confuses non-technical users, leaks internal state.
- **Size:** small (delete four lines, keep `console.log`)
- **Beta impact:** medium — first impression polish
- **Dependencies:** none

### 6. Duplicate function definitions

- **What:** [obNextSkip2 defined at 4813 AND 4835](../index.html:4813), [obBackSkip2 defined at 4825 AND 4847](../index.html:4825). Second definition overwrites the first; HTML only references one pair. [renderLifeGoalsEditor:4579](../index.html:4579) + `updateLifeGoal` exist but the `#lifeGoalsEditor` DOM node doesn't — dead code.
- **Problem:** Confusing on read. Dead code implies settings has a "goals" section that was cut but never cleaned up.
- **Size:** small
- **Beta impact:** low — not user-visible
- **Dependencies:** none

### 7. Realtime polling burns Supabase budget

- **What:** [setupRealtime:1897](../index.html:1897) `setInterval` every 8 seconds, no visibility-aware gating, no stop on partner-view switch beyond an early-return.
- **Problem:** ~10,800 anon-role Supabase reads per user per day, most for a hidden tab or background PWA. Burns the free-tier quota once 5 couples join, and any Supabase outage re-tries hard.
- **Size:** small — gate on `document.visibilityState === 'visible'`, or better, swap to Supabase realtime channels.
- **Beta impact:** medium — quiet until it bites, then it bites hard
- **Dependencies:** none

### 8. `renderTodayCard` diverges from every other task card

- **What:** [renderTodayCard:3600](../index.html:3600) renders a completely separate card shape: no `priCardClass`, no `dueBadgeHTML`, due label is hardcoded raw YYYY-MM-DD (`Due 04/23/26`), no AI-Complete/Assist action buttons, no edit-vs-delete parity. The other three render paths (`renderHatTasks`, `showDrilldown`, `showLifeDrilldown`) share 80% of the card template.
- **Problem:** The same overdue task looks different on the Today tab than on its project tab. No red tint, no ⚠️ label, raw ISO date. Undoes everything just shipped for overdue visibility.
- **Size:** small — extract card renderer and share it
- **Beta impact:** medium — Today is a core surface
- **Dependencies:** pairs with observation #2; if Today is redesigned anyway, fold this in

### 9. Partner-view is half-wired

- **What:** [switchView:5491](../index.html:5491) swaps `tasks` + `userHats` when viewing partner, but doesn't re-run `loadKnowledge`, `loadCorrections`, `loadUserLifeCategories`, or re-render Dashboard/Today/Journey. The side-panel indicator says "Viewing partner's tasks and life score" but the Life Score on the Dashboard is still the user's own. Chat context ([sendChat:5076](../index.html:5076)) uses the swapped `tasks` list with the viewer's `ai_context`, so Claude talks to Mia about Vijay's tasks using Mia's personal background. Strange cross-wiring.
- **Problem:** The partner promise ("see each other's world") is a surface veneer over shared mutable state. Power users will notice Dashboard numbers don't change.
- **Size:** medium — requires a proper view-owner concept instead of swap-globals
- **Beta impact:** medium — partner linking is a core differentiator per positioning statement
- **Dependencies:** none

### 10. `renderStats` re-fetches completed tasks on every trigger

- **What:** [renderStats:3058](../index.html:3058) is invoked from `showView('stats')`, `toggleDone`, `quickComplete`, `bulkMarkComplete`, `removeTask`, `setupRealtime`, and indirectly from `renderDrilldownAfterToggle`. Each call makes an uncached Supabase fetch for last-30-days completed tasks and an uncached Claude call for the AI advisor (advisor has a fingerprint cache, the fetch doesn't).
- **Problem:** Rapid-fire toggles (bulk complete of 5 tasks) trigger 5 renderStats → 5 completed-task fetches. Wasteful and makes the UI feel jittery.
- **Size:** small — memoize completed-tasks for the same fingerprint or debounce
- **Beta impact:** low — not a user-visible bug, but a latency tax that accumulates
- **Dependencies:** none

### 11. `resend_key` column dual-purposed for ChatGPT key

- **What:** [saveApiKey:3878](../index.html:3878) stores a ChatGPT key in `profile.resend_key` with comment `"reuse resend_key col for chatgpt key temp storage"`.
- **Problem:** `resend_key` is read by the digest pipeline (conceptually — actual code uses `process.env.RESEND_API_KEY`, so this is currently a dead write, but the column name lies). A future engineer who actually reads `resend_key` expecting a Resend key will hit ChatGPT keys. Schema integrity trap.
- **Size:** small — add a proper `openai_key` column, migrate, drop the comment
- **Beta impact:** low — most users pick Claude
- **Dependencies:** schema migration

### 12. OAuth requests Calendar scope but UI is DevTools-only

- **What:** [signInWithGoogle:1836](../index.html:1836) requests `https://www.googleapis.com/auth/calendar.readonly`. [fetchCalendarPrimary:1802](../index.html:1802) is invokable from DevTools only — no UI entry point. Token cached in `sessionStorage` (clears on tab close; every PWA cold start = expired).
- **Problem:** Users approve a Calendar permission they never see used. When Phase 2 digest integration lands, `sessionStorage` token approach forces a re-auth on every PWA launch.
- **Size:** small for now (store in `localStorage` with expiry), medium for Phase 2 (proper refresh flow via `access_type: 'offline'` server-side)
- **Beta impact:** medium — consent friction on sign-up, and the Calendar NOW item is scoped for this season
- **Dependencies:** coordinate with Calendar Phase 2 scope

### 13. No service worker / offline story for a PWA

- **What:** Manifest-style meta tags present ([index.html:6-9](../index.html:6)), `isPWA` detection in place, but no `<link rel="manifest">`, no `navigator.serviceWorker.register()`, no cached shell.
- **Problem:** On spotty wifi (kids' activities, car, coffee shop — the exact couples-beta context), every Supabase blip shows an error toast and the app is unusable. For a mobile-first household OS this is a credibility hole.
- **Size:** large — real SW, manifest, offline mutations queue
- **Beta impact:** medium — invisible when things work; kills trust when they don't
- **Dependencies:** no framework means rolling SW by hand

### 14. Life-score math whipsaws on small task counts

- **What:** [renderStats:3174-3177](../index.html:3174) computes `neglectPenalty = min(40, neglectedCount * 8)` and `concentrationPenalty = 15 if maxShare > 60 else 8 if > 40`. With 3 tasks all in Work → 100% concentration + 6 neglected = score 45. Adding one task to any other category swings the score 10+ points.
- **Problem:** Week-1 couples will see scores ping-ponging for no clear reason; the dashboard's headline number feels unstable rather than diagnostic. The digest uses a different formula ([digest.js:121-135](../api/digest.js:121)) — goal-based, not share-based — so the two scores never agree.
- **Size:** medium — unify the two formulas and add a "not enough data yet" gate until N≥~15 tasks/week
- **Beta impact:** medium — Life Score is the headline dashboard number and the hero email stat
- **Dependencies:** consider whether to deprioritize the life-score altogether (CLAUDE.md backlog already flags it)

### 15. Onboarding "Your AI" step hidden behind Skip2 stub

- **What:** [onboarding step 2 at 882](../index.html:882) exists in HTML (Claude vs ChatGPT picker) but [obNextSkip2:4835](../index.html:4835) unconditionally jumps from step 1 → step 3. The button never fires. Dots show 5 steps. Step 2 HTML is 20 lines of orphan markup.
- **Problem:** Hosted-by-STWRD is the right non-technical default, but the leftover markup and skip2 function names are a maintenance landmine. If a future engineer restores step 2 they'll find two definitions of each skip function.
- **Size:** small — delete step-2 HTML and the `obNextSkip2` duplicate
- **Beta impact:** low — not user-visible
- **Dependencies:** none

### 16. `clearAll` uses browser `confirm()`, deletes everything with one click

- **What:** [clearAll:3900](../index.html:3900) — native `confirm()` modal, then `DELETE FROM tasks WHERE user_id`.
- **Problem:** No undo, no archive, no "type CLEAR to confirm". One accidental tap wipes a couple's entire task history (and by extension their life-score trend data). Native `confirm` dialogs look broken on iOS PWA.
- **Size:** small — in-app confirmation with typed gate, or soft-delete
- **Beta impact:** medium — low probability, catastrophic blast radius
- **Dependencies:** none

### 17. Digest send-time claim drift (6am / 7am / cron-job.org / Vercel cron)

- **What:** [api/digest.js header comment](../api/digest.js:1) says 6am CT. [Settings UI:1267](../index.html:1267) says 6am CT. [Onboarding step 1:866](../index.html:866) and [step 6:1009](../index.html:1009) say 6am CT. CLAUDE.md says 7am CT. The actual trigger is cron-job.org on a schedule that isn't in the repo, plus Vercel cron status per CLAUDE.md is "unverified."
- **Problem:** Four different sources of truth for when the most important daily user touchpoint fires. Beta guide PDF probably says a fifth thing.
- **Size:** small — pick one time, update in-app copy, document the cron owner
- **Beta impact:** low — users mostly don't check the clock, but shows drift
- **Dependencies:** confirm Vercel cron is off (CLAUDE.md open question)

## Observations that aren't actionable but worth noting

- **Safari-workaround sprawl.** [initApp:1517](../index.html:1517), [loadProfile:1708](../index.html:1708), [sbFetch:1918](../index.html:1918), [completeOnboarding:4903-4923](../index.html:4903) all bypass the Supabase SDK by reading from `localStorage` directly because the SDK hangs on Safari's auth lock. This works but the auth path has 4+ different ways to get a token, a no-op `lock` override ([index.html:1479](../index.html:1479)), manual PKCE exchange ([1531-1588](../index.html:1531)), direct token refresh ([1772](../index.html:1772)), and a `candidateKeys` fallback list in onboarding. One focused refactor could collapse this to a single `getValidToken()` helper — but doing it piecemeal risks regressing the 2026-04-20 sign-in fix.
- **Hat / project naming drift.** Variable names toggle between `hat` and `project` across 40+ functions (`openHat`, `userHats`, `renderHatBuckets`, but DB table `projects` and column `project`). Evidence the product was renamed and code wasn't. Not a bug, just tax on every code read.
- **Three overlapping correction pipelines.** `corrections` is written by [saveEdit:3519](../index.html:3519), [bulkMoveToProject:3004](../index.html:3004), and surfaced to the prompt in [processTask:2096](../index.html:2096). CLAUDE.md feedback memory flags "AI behavior changes need test cases" — the classifier's learning loop has no evaluation harness. Before tuning the prompt, a ~15-task golden set is the right prerequisite; the feedback memory already flags this.
- **Accessibility is roughly absent.** No `aria-label` on icon-only buttons (mic, close, chat FAB). No visible focus rings. Color contrast of `--text2 (#8888aa)` on `--surface (#12121a)` is ~4.3:1 — WCAG AA for large text only. Most priority/overdue signaling is color-only; overdue adds a ⚠️ emoji but P1 vs P2 is color alone. Not a near-term beta blocker but becomes one the day someone with low vision tries the app.
- **`knowledgeBase` and `corrections` are not cleared on partner switch.** When viewing Mia's tasks, `processTask` would run with Vijay's project knowledge context — except `processTask` is disabled in partner view by normal UX flow (no capture bar). So currently benign, but a latent bug if capture ever opens in partner view.
- **Editing a task kills scroll position.** Every `innerHTML = …` in `renderHatTasks`, `renderStats`, etc. re-renders the whole list. On a project with 40+ tasks, toggling one sends the scroll to top. Noticeable on phone.
- **`chatBtn` z-index overlaps `bulkActionsBar`.** Both at z-index 150 ([5629](../index.html:5629), [1330](../index.html:1330)). In bulk-select mode with chat open, stacking is undefined. Minor but the kind of thing that surfaces on one specific device.
- **Route/URL state is near-zero.** Nav selection, open project, drilldown, selected life area — none persist in the URL. Browser back on mobile exits the app instead of backing out of a drilldown. Expected cost of single-file architecture; noting it because couples sharing links of "this task" won't work.
- **`about` settings line reads "STWRD v2.0".** Line [1323](../index.html:1323). Not connected to anything. Will get stale the day someone cuts v2.1.
